How memory actually works
You start a session
Pick a folder in Cowork
→
Claude reads CLAUDE.md
In the selected folder
→
Claude reads memory/
Sub-files with details
→
Session runs
Claude knows your context
→
Session ends
Memory persists in files

How to Use Claude Right

Your personalized playbook based on your actual sessions and setup.

The Right Mental Model
Claude Cowork is not a chat window. It's a co-worker with file access and connected tools.
Think of it like: you give your co-worker (Claude) a folder to work out of, they read your notes (CLAUDE.md), and they can take real actions in Stripe, Vercel, Make, and Chrome without you doing it yourself. Every session, they catch up by reading your files. When the session ends, nothing is lost — it's all in the files.
Your Ideal Session Flow
1
Open Cowork with your Obsidian vault selected
Not the project folder. The vault. Claude reads CLAUDE.md and immediately knows who you are, your clients, your stack, and your preferences.
2
State one clear task
"Build the hero section for Somewhere's retainer email" or "Invoice Pierre $500 deposit via Stripe." One task per session keeps context tight and tokens efficient.
3
Let Claude use its connectors and skills
Don't manually copy code into Cursor if Claude can write and save the file directly. Don't log into Stripe if Claude has the connector. Don't check Vercel if Claude can read the build logs.
4
When the task is done, start a new session
Claude will nudge you. Fresh session = clean context = faster, cheaper, less likely to get confused. Your CLAUDE.md carries everything forward.
